幻觉预防与可靠性提升

Lesson 8 · 让 AI 的回答更可信 · 约 25 分钟

所有大模型都会产生幻觉（hallucination）——编造事实、虚构引用、自信地给出错误答案。 好消息是：大部分幻觉不是模型"笨"，而是你的 prompt 给了它幻觉的空间。

本课整合前面几课中的防幻觉技巧（引用先行、自检、CoT），并补充新的策略， 形成一套完整的可靠性提升体系。

► 官方参考：Anthropic 官方教程 — Ch8: Avoiding Hallucinations

Anthropic 在新模型上大幅减少了幻觉，但不可能完全消除。关键是：不要期望模型"不说谎"，而是通过 prompt 设计让它难以说谎。

幻觉的三种类型

类型示例主要原因
事实错误"2024 年奥运会北京举办"模型知识边界外，或训练数据偏差
虚构引用编造论文标题和作者模型"填空"本能——它觉得这里应该有引用

技巧一：给 AI 说"不知道"的权利

很多幻觉源自模型觉得"必须给个答案"。如果你明确允许它说不知道，幻觉率大幅下降：

✗ 隐含"必须答"
2024 年 WWDC 上 Apple 发布了哪些新 AI 功能？
✓ 允许不知道
2024 年 WWDC 上 Apple 发布了哪些新 AI 功能？ 如果你不确定或记忆模糊，请明确说"我不确定"而不是猜测。

在 system prompt 中也可以加上：

如果你不确定答案，说"我不确定"而不是虚构。
这对于日期、统计数据、引用来源等问题尤其重要。

技巧二：引用先行（防幻觉第一利器）

从 L5 回顾并强化：当处理长文档时，强制 AI 先引用原文再分析。

<task>分析这份合同中的潜在风险条款。</task>

<contract>（合同全文）</contract>

在分析之前，先从合同中找出与风险相关的原文引用，
放在 <quotes> 标签中。然后基于这些引用分析风险。

如果合同中没有明确提及某个风险，不要说"可能存在风险"。
只有在有原文依据时才做分析。
关键机制：引用先行打断了模型"先猜测后找证据"的路径，强制它"先找证据再下结论"。这是单次 prompt 中最有效的防幻觉手段。

技巧三：内部自检

在 prompt 末尾追加自检指令——让 AI 在给出最终答案前验证自己的输出：

在结束之前，请检查你的回答：
1. 所有引用的数据是否都来源于上面给出的文档？
2. 结论是否有明确的推理步骤支撑？
3. 是否有任何陈述是你"觉得"但无法从材料中证实的？

如果发现不准确之处，在 <corrections> 中修正。

自检在编码任务中尤其有效：

写一个 Python 函数，检测字符串是否回文。

完成代码后，在 <verification> 中列出你对代码的检查结果：
- 边界条件是否处理了（空字符串、单字符、大小写）？
- 是否有性能问题？
- 是否有 Python 版本兼容性问题？

技巧四：降低 prompt 本身的模糊性

模糊的 prompt 是幻觉的温床——模型不知道你要什么，于是"猜"一个答案。

模糊写法明确写法
分析这份数据分析这份数据，回答三个问题：1. 增长率是多少？2. 主要增长来源是什么？3. 有什么异常点？
帮我写个方案帮我写一个 3 个月的敏捷转型方案，包含阶段划分、里程碑和风险评估
这代码有什么问题审查这段代码：检查潜在 bug、安全隐患和性能瓶颈，对每个问题标注严重程度

技巧五：链式验证

结合 L7 的链式技巧，用多步验证来排除幻觉：

生成答案 → 事实核查 → 修正输出

Step 1 — 生成：正常执行任务，但标注不确定的点。

生成方案。对任何你不确定的数据或假设，用 \[不确定\] 标注。

Step 2 — 事实核查：专门检查上一步中的不确定点。

以下方案中有多处 \[不确定\] 标注。请逐一核查：
1. 哪些有可靠来源支持？
2. 哪些需要进一步调研？
3. 哪些是明显错误需要删除？

Step 3 — 修正：基于核查结果输出最终版本。

技巧六：temperature 与 effort 控制

在 API 层面，幻觉与创造性正相关：

参数低值高值
temperature0（几乎无随机性，适合事实型任务）0.7+（适合创意写作，但幻觉更多）
Claude effortlow（快速回答，较少"多想"出来的幻觉）max（深入思考，但可能产生"想太多"的幻觉）

经验法则：事实提取、分类、结构化输出 → temperature = 0。创意写作、头脑风暴 → temperature 0.5–0.7。

✍️ 练习：为法律合同审查设计防幻觉 prompt

你需要 AI 审查一份合同，找出潜在风险。合同有 5000 字。请设计一条 prompt，要求 AI 引用原文、标注不确定点、自检，且不得虚构条款。

提示：结合引用先行（L5）、允许说不知道（本课）、自检（L3）。
<task>审查以下合同，找出潜在风险条款。</task>

<contract>（合同全文）</contract>

<rules>
1. 对每个风险点，必须先引用原文再分析
2. 如果对某条款的解读不确定，标注 \[需确认\] 而不是猜测
3. 不得虚构不存在的条款或常见的"标准条款"
4. 完成分析后，在 <self_check> 中检查是否所有结论都有原文依据
</rules>

跨模型适用性

技巧所有模型说明
允许说"不知道"★★★所有模型都倾向于"给答案"，明确允许不知道能减少幻觉
引用先行★★★强制模型基于证据而非记忆回答
内部自检★★★给模型一次"纠正自己"的机会
降低 prompt 模糊性★★★模糊输入→模糊输出，是所有模型的通病
temperature=0★★★所有 API 都支持，对事实型任务至关重要
跨模型结论：防幻觉技术几乎全部通用。唯一模型相关的是内部 thinking 机制——Claude 的 adaptive thinking 可以在不占用输出 token 的情况下做自检，而其他模型需要通过 prompt 显式要求。

总结

  1. 允许说"不知道" — 最简单的防幻觉手段，只需一句话
  2. 引用先行 — 先找证据再下结论，单次 prompt 最有效
  3. 内部自检 — 让 AI 在交付前检查自己的输出
  4. 降低模糊性 — 模糊 prompt = 幻觉的温床
  5. 链式验证 — 生成 → 事实核查 → 修正
  6. temperature=0 — 事实型任务永远设为 0

课后行动：在日常工作中找一个你经常遇到幻觉的场景，应用"引用先行+允许不知道+自检"组合拳，对比前后效果。
