长上下文提示词工程

Lesson 5 · 处理大规模文档 · 约 25 分钟

现代大模型支持数十万 token 的上下文窗口。但"能放进"和"能处理好"是两回事。 Anthropic 的研究发现：prompt 结构对长上下文的输出质量有高达 30% 的影响。

本课聚焦一个问题：当你有一份（或多份）长文档要交给 AI 分析时，如何组织 prompt 才能获得最佳结果。

► 官方参考：Anthropic — Long context prompting

Anthropic 测试表明：将查询放在文档后面（而不是前面），回答质量提升高达 30%，尤其是在复杂、多文档输入场景中。

核心原则：文档在上，查询在下

这是长上下文 prompt 最重要的布局规则：

<documents>
<document>...（长文档内容）...</document>
<document>...（第二份文档）...</document>
</documents>

（在这里放你的指令和查询）

为什么有效？ 模型在处理长文档时，靠近输入末尾的内容保留度更高。把文档放前面、查询放后面，相当于让模型"先读材料，再听问题"。

✗ 查询在前
分析这份年报中关于 AI 战略的部分。 {{这里粘贴了 5000 字的年报全文}}
✓ 文档在前
<documents> <document> {{年报全文}} </document> </documents> 分析这份年报中关于 AI 战略的部分。

多文档结构：带元数据的 XML

当有多份文档时，给每份文档加上元数据：

<documents>
  <document index="1">
    <source>annual_report_2025.pdf</source>
    <document_content>
      {{年报全文}}
    </document_content>
  </document>
  <document index="2">
    <source>q1_earnings_call.txt</source>
    <document_content>
      {{Q1 财报电话会议记录}}
    </document_content>
  </document>
</documents>

对比年报中的战略方向与 Q1 电话会议中的实际执行情况，找出一致和矛盾之处。

元数据的好处：

  • AI 可以在回答中引用具体文档来源
  • 不同类型的内容清晰区分
  • 可以按文档维度做对比分析

引用先行：用 quotes 提升准确率

这是长上下文场景中最被低估的技巧。让 AI 先引用原文再分析，相当于强制它"找到证据再下结论"，大幅减少幻觉。

在处理长文档时，先让 AI 引用相关原文，再做分析：

<task>
你是一名 AI 医生助理。帮助医生诊断患者可能的疾病。
</task>

<documents>
  <document index="1">
    <source>patient_symptoms.txt</source>
    <document_content>{{症状}}</document_content>
  </document>
  <document index="2">
    <source>patient_records.txt</source>
    <document_content>{{病历}}</document_content>
  </document>
</documents>

从病历和症状记录中找出与患者症状相关的引用，放在 <quotes> 标签中。
然后基于这些引用，列出对诊断有帮助的所有信息，放在 <info> 标签中。
实用场景：合同审查、论文审稿、PRD 分析、代码审查——任何需要"从长文档中提取证据"的任务都非常适合引用先行。

长上下文的"分层"策略

当文档非常长（10万 token 以上）时，单次分析可能不够。用分层策略：

层级做法效果
第一层：粗读让 AI 先整体扫描文档结构，给出摘要和章节划分建立全局认知
第二层：精读针对特定章节深入分析深入细节
第三层：交叉跨章节/跨文档对比分析发现矛盾与关联

直接在 prompt 中描述这个分层过程：

请按以下步骤分析这份文档：

第一步：整体扫描。给出文档的全局摘要和主要章节划分。放在 <overview> 中。

第二步：针对 [特定主题]，从文档中找到所有相关段落，引用原文后分析。放在 <deep_dive> 中。

第三步：如果不同章节对同一问题的表述有矛盾，指出矛盾所在。放在 <conflicts> 中。

长上下文的实用技巧

  1. 引用先行防幻觉 — 先引用原文再分析，是长上下文场景最有效的防幻觉手段
  2. 文档在上查询在下 — 文档放在 prompt 顶部，查询和指令放在底部
  3. 元数据标注 — 给每份文档标注来源、日期、索引，便于交叉引用
  4. 分层处理 — 超长文档用粗读→精读→交叉分析的分层策略
  5. 限制范围 — 不要问"分析整份文档"，而问"分析文档中关于 X 的部分"
✍️ 练习：设计多文档分析 prompt

你有 3 份文档要分析：一份 PRD、一份技术方案、一份测试报告。请设计一个 prompt 让 AI 对比它们，检查技术方案是否完整覆盖了 PRD 需求，测试报告是否覆盖了所有功能点。要求引用原文后再分析。

提示：用 <documents> 结构化，先引用再分析，查询放在最后。
<task>
对比 PRD、技术方案和测试报告，检查覆盖度。
</task>

<documents>
  <document index="1"><source>PRD.md</source><document_content>{{PRD}}</document_content></document>
  <document index="2"><source>技术方案.md</source><document_content>{{方案}}</document_content></document>
  <document index="3"><source>测试报告.md</source><document_content>{{报告}}</document_content></document>
</documents>

分析步骤：
1. 从 PRD 中提取所有功能需求，放在 <requirements> 中
2. 从技术方案中找到对应每个需求的设计，放在 <coverage> 中
3. 从测试报告中找到验证每个功能的用例，放在 <testing> 中
4. 列出未被覆盖的需求和未测试的功能，放在 <gaps> 中

每个步骤都引用原文作为证据。

总结

  1. 文档在上，查询在下 — 提升回答质量最高达 30%
  2. XML 结构化 — 多文档用 <documents><document> 包裹，标注元数据
  3. 引用先行 — 先引原文再分析，防幻觉最佳手段
  4. 分层策略 — 粗读 → 精读 → 交叉分析
  5. 限制范围 — 聚焦具体问题而非泛泛分析

课后行动：下次处理长文档分析任务时，试试"文档在上+引用先行"，看看输出质量的变化。你会发现幻觉明显减少。
