工具调用与 Agent 系统

Lesson 9 · Tool Use & ACI · 约 25 分钟

之前的课程都假设 AI 只能回答。但真实应用中，AI 需要行动——查数据库、发请求、改文件。 这就是工具调用（Tool Use / Function Calling）。

Anthropic 提出了 ACI（Agent-Computer Interface） 的概念——你把工具定义写得越清晰， Agent 就越少犯错。写工具定义的投入，和写人机界面的投入一样重要。

► 官方参考：Anthropic — Tool use · Building Effective Agents

Anthropic 在 SWE-bench 上的经验：花在优化工具定义上的时间比优化 system prompt 更多。工具定义就是 Agent 的"用户界面"。

工具调用的本质

工具调用不是"AI 学会了一个新功能"，而是：

  1. 你给 AI 一份工具清单（名称、参数、用途）
  2. AI 决定"这个问题需要调用工具 X"
  3. AI 输出一个结构化的工具调用请求
  4. 你的代码执行工具，把结果返回给 AI
  5. AI 基于工具结果继续推理

整个流程是思考 → 行动 → 观察 → 再思考的循环。

写优秀工具定义的四个原则

1. 设身处地为模型想

工具描述是否足够清晰？换成一个初级开发者能正确使用吗？

// 差的定义
"搜索文档"

// 好的定义
"搜索知识库文档。用关键词和过滤条件找到相关文档。
- 关键词越具体，结果越精确
- 支持 AND/OR 组合
- 如果结果太多，建议用户缩小范围"

2. 防呆设计（Poka-yoke）

让参数设计使错误难以发生。Anthropic 的案例：

// 问题：Agent 使用相对路径导致混乱
// 修复：将工具改为强制要求绝对路径
"参数 filepath: 要求绝对路径，例如 /home/user/projects/foo/src/main.ts"

3. 提供示例和边界情况

{
  "name": "calculate_discount",
  "description": "计算商品折扣价。支持阶梯折扣和叠加优惠。",
  "parameters": {
    "price": { "type": "number", "description": "原价，单位元" },
    "discount_percent": { "type": "number", "description": "折扣百分比，0-100" },
    "vip_discount": { "type": "number", "description": "VIP 额外折扣，0-100，可选" }
  },
  "examples": [
    { "price": 100, "discount_percent": 20, "result": 80 },
    { "price": 200, "discount_percent": 10, "vip_discount": 5, "result": 171 }
  ]
}

4. 有限但够用

不是工具越多越好。每个多余的工具都会增加模型选错的概率。

ACI 黄金法则：把工具定义当作 API 文档来写——清晰、完整、有例子。模型是你的"API 调用者"，它需要知道每个工具做什么、什么时候用、怎么用。

引导模型使用工具

新 Claude 模型默认更主动。如果你想让 AI 更积极地使用工具：

<tool_guidance>
默认使用工具来完成任务，而不是只给建议。
如果用户意图不明确，推断最可能需要的工具操作并执行。
</tool_guidance>

如果你想让 AI 更谨慎：

<tool_guidance>
只在明确必要时才调用工具。可以先给出分析和建议，
等用户确认后再执行具体操作。
</tool_guidance>

并行工具调用

Claude 新模型擅长并行工具调用——比如一次发起多个搜索、读多个文件。这可以大幅加速 Agent 任务。

如果工具之间没有依赖关系，让 AI 并行调用：

<use_parallel_tool_calls>
如果要调用多个没有依赖关系的工具，同时并发执行。

例如：需要读 3 个文件时，同时发 3 个读文件请求。
但如果 B 工具需要 A 的结果作为参数，则必须串行。
</use_parallel_tool_calls>

✍️ 练习：设计一个代码审查工具的 ACI

你需要定义一个工具，让 AI 可以审查代码文件。该工具应接受文件路径和审查范围，返回问题列表。请写出工具定义（名称、描述、参数）。

提示：参考 ACI 四原则——清晰描述、防呆设计、提供示例。
{
  "name": "review_code",
  "description": "审查指定代码文件，返回问题列表。文件路径必须为相对于项目根目录的路径。支持指定审查范围以缩小焦点。",
  "parameters": {
    "file_path": {
      "type": "string",
      "description": "文件路径（相对项目根目录），如 src/utils/auth.ts"
    },
    "scope": {
      "type": "string",
      "enum": ["all", "security", "performance", "bug", "style"],
      "description": "审查范围。all = 全部，其他值只审查特定类型问题。默认 all。"
    }
  },
  "examples": [
    { "file_path": "src/api/handler.ts", "scope": "security", "result": "发现 2 个安全问题..." }
  ]
}

总结

  1. 工具调用 = 思考→行动→观察 循环
  2. ACI 四原则：设身处地、防呆设计、提供示例、有限但够用
  3. 主动 vs 保守：用 tool_guidance 控制 AI 的工具使用倾向
  4. 并行调用：无依赖的工具同时执行可大幅提速
  5. 跨模型：所有主流模型都支持 function calling，ACI 原则通用
