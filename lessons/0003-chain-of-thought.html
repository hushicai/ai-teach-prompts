思维链与推理 — 让 AI 先想后答

Lesson 3 · Chain of Thought & Thinking · 约 25 分钟

前面两节课我们学了如何描述任务（L1）和展示任务（L2）。本课要学第三个关键维度：让 AI 在回答前先思考。

对于复杂问题，直接问会让 AI 给出一个"直觉性"的回答——这在简单任务上够用，但在数学、逻辑、多步推理任务上会出错。 思维链（Chain of Thought, CoT） 就是让模型把推理过程一步一步写出来，然后再给出最终答案。

► 官方参考：Anthropic — Thinking and reasoning

Anthropic 在官方教程中将 CoT 称为 "Precognition"（预认知）——让模型在回答前先预见推理路径。

为什么"先想后答"有效？

想象一下：你让一个实习生分析季度销售数据。如果他直接说"Q3 业绩不错"，你没法知道他是否真的分析了数据。但如果他说：

"让我一步步看：第一，Q2 到 Q3 总销售额增长了 15%；第二，增长主要由新品线驱动（占比 72%）；第三，老品线下降了 8%；第四，考虑到老品线的下降趋势，Q4 建议..."

这个推理过程让你能判断他的分析质量，也让他自己不易出错。CoT 的原理相同——强制模型暴露推理路径，减少"跳跃式"错误。

两种模式：Prompt CoT vs Thinking

特性Prompt CoT（通用）Thinking（Claude 原生）
原理在 prompt 中要求"分步思考"模型在内部生成推理 token
输出可见用户可见推理过程推理过程在 API 的 thinking block 中，可选择是否暴露
适用模型所有大模型Claude Opus 4.6+, Sonnet 4.6+
控制粒度手动（在 prompt 中控制）effort 参数（low/medium/high/max）
适用场景所有需要推理的任务复杂 agent 任务、长链推理

模式一：Prompt CoT（通用，所有模型可用）

这是最基础的 CoT，适用于任何模型。核心就是一句话："在给出最终答案前，请一步步思考。"

✗ 直接问
一个商店卖苹果，进价 5 元，售价 8 元。今天卖了 120 个苹果，但退货了 5 个。今天总利润多少？
✓ 加思维链
一个商店卖苹果，进价 5 元，售价 8 元。今天卖了 120 个苹果，但退货了 5 个。今天总利润多少？ 在给出最终答案前，先一步步推理。

进阶：用结构化标签隔离推理与答案

<question>
一个商店卖苹果，进价 5 元，售价 8 元。今天卖了 120 个苹果，但退货了 5 个。今天总利润多少？
</question>

<reasoning>
（一步步推理，不要直接给出答案）
</reasoning>

<answer>
（基于 reasoning 给出最终答案）
</answer>
使用 <reasoning> 和 <answer> 分离的好处：你可以在代码中分别提取推理过程和最终答案，或将推理过程对用户隐藏而只展示答案。

在 few-shot 示例中加入 CoT

你可以在 few-shot 示例中展示"推理过程→最终答案"的模式，让模型学习这个格式：

<task>
回答数学应用题。先用 <reasoning> 写出解题步骤，再用 <answer> 给出答案。
</task>

<examples>
<example>
<question>小明有 15 个苹果，给了小红 7 个，又买了 10 个。他现在有多少个？</question>
<reasoning>
一开始 15 个。给了小红 7 个：15 - 7 = 8 个。又买了 10 个：8 + 10 = 18 个。
</reasoning>
<answer>18 个</answer>
</example>
</examples>
✏️ 测验 1：以下哪条 prompt 最有效地应用了 CoT？

模式二：Claude 原生 Thinking（高级）

Claude Opus 4.6+ 使用 adaptive thinking（自适应思考），模型自动决定何时思考、思考多久。通过 effort 参数调节深度：low / medium / high / max。

在 Claude API 中启用 Thinking 后，模型会在内部生成推理 token，然后给出可见回答。好处是：

  • 推理不占用输出 token 预算
  • 模型可以"在心里"进行更复杂的推理
  • 通过 effort 参数精确控制思考深度

如果你正在用 Claude（通过 API 或某些客户端），可以这样引导它的思考行为：

在收到工具结果后，仔细评估它们的质量，然后确定最佳下一步。
用你的思考能力来规划和迭代，然后执行最佳行动。

或者，如果你想限制过度思考：

深思会增加延迟，只有在真正能提升回答质量时才使用——通常是需要多步推理的问题。
不确定时，直接回答。
跨模型提示：GPT-4o 的 reasoning_effort 参数和 Gemini 的 thinking 功能类似。但 CoT（通过 prompt 要求分步思考）是唯一所有模型通用的技巧——优先掌握它。

自检：让 AI 验证自己的答案

Anthropic 推荐在 prompt 结尾加一句自检指令，在编码和数学任务上效果显著：

在结束之前，请对照[检查标准]验证你的答案。

或者在 CoT 框架中嵌入验证步骤：

<reasoning>
（写出推理步骤）
</reasoning>

<verification>
（检查 reasoning 中每一步是否正确，是否有遗漏）
</verification>

<answer>
（确认无误后的最终答案）
</answer>
✍️ 练习 1：为数据分析任务设计 CoT prompt

你需要让 AI 分析一份销售数据并给出建议。请设计一个 prompt，要求 AI 先推理再回答，同时将推理与答案分离。

提示：用 <task> 描述任务，用 <reasoning> 要求分步分析，用 <answer> 给出最终建议。可以加上 <verification> 自检步骤。
<task>
分析以下销售数据，给出 Q3 运营建议。
</task>

<data>
Q2 销售额: ¥1,200,000
Q3 销售额: ¥1,500,000
新客户占比 Q2: 30%
新客户占比 Q3: 45%
客单价: ¥320
退货率: 8%
</data>

<reasoning>
先分析各项数据的变化趋势，找出增长的主要驱动力，再识别潜在风险。不要跳过任何步骤。
</reasoning>

<verification>
检查 reasoning 是否覆盖了所有数据维度，逻辑是否自洽。
</verification>

<answer>
基于 reasoning 给出 3 条最关键的运营建议。
</answer>

总结

  1. CoT 是通用技术 — 所有模型都支持"分步思考"的 prompt
  2. 结构化分离 — 用 <reasoning> 和 <answer> 标签隔离推理过程与最终输出
  3. 在示例中加入推理 — few-shot 示例也可以展示"推理→答案"的完整链路
  4. 自检提高准确率 — 在 CoT 末尾加 verification 步骤，尤其适合编码和数学
  5. 适应性思考 — Claude 用户可以通过 effort 参数精确控制推理深度
课后行动：找一个你平时需要 AI 做"多步推理"的任务（分析报告、代码调试、决策建议），用 CoT prompt 重写，观察输出质量的变化。
