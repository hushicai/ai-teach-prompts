Prompt Chaining — 多步提示词链

Lesson 7 · 分解复杂任务 · 约 25 分钟

前几课所有的技巧都关注单次 prompt。但很多真实任务太复杂，一次对话搞不定—— 比如"分析财报 → 提炼关键指标 → 写摘要 → 翻译成英文"。这时候需要 Prompt Chaining。

► 官方参考：Anthropic — Chain complex prompts

Anthropic 指出：新模型（Opus 4.6+）的 adaptive thinking 可以在单次内部做复杂推理。链式调用在需要检查中间结果或强制特定流水线时仍然是最好的选择。

为什么要链式调用？

单一 prompt 的局限：

  • 注意力稀释 — 任务越长、步骤越多，AI 越容易遗漏或混淆
  • 无法检查中间结果 — 如果中间步骤错了，最终输出不可能对
  • 调试困难 — 你不知道是哪一步出了问题
  • token 浪费 — 如果第一步就出错，后面的推理白做了

链式调用把一个长任务拆成多个短任务，每一步聚焦一个目标：

提取数据 → 分析趋势 → 撰写报告 → 翻译输出

模式一：自纠正链（Generate → Review → Refine）

这是最实用、最通用的链式模式，分为三步：

Step 1: 生成（Generate）

<task>根据以下需求，写一封面向客户的版本更新邮件。</task>
<context>（产品更新说明）</context>

Step 2: 审查（Review）

<task>审查以下邮件是否符合标准：
1. 语气是否适合非技术客户？
2. 是否突出了客户能感知到的价值？
3. 是否有不明确的缩写？
4. 是否有不必要的技术细节？

列出每个问题的具体问题。</task>
<draft>（上一步的输出）</draft>

Step 3: 优化（Refine）

<task>根据审查反馈优化这封邮件。只修改有问题的地方，保留好的部分。</task>
<feedback>（上一步的输出）</feedback>
<draft>（第一步的输出）</draft>
适用场景：邮件/文案写作、代码审查、翻译润色、内容审核——任何需要"先产出再优化"的任务。

模式二：分解链（Decompose → Execute → Assemble）

适用于大型分析任务：

// Step 1: 分解
分析这个需求，列出所有需要实现的独立功能模块。
按依赖关系排序。

// Step 2: 执行（对每个模块分别执行）
实现模块 X。输入是 [上一步的输出]，输出是代码和测试。

// Step 3: 组装
将所有模块整合。检查接口是否匹配，测试是否覆盖完整链路。

什么时候链式、什么时候单次？

适合链式适合单次
任务有清晰的阶段划分任务简单直接
需要检查中间结果一次推理就能完成
不同阶段需要不同角色/指令不需要切换上下文
单次 prompt 超过 4k tokenprompt 简洁明了
你在调试/迭代某个复杂工作流已稳定的流水线
经验法则：如果你发现自己在一个 prompt 里写了很多"第一步...第二步...第三步..."，这就是一个信号——该拆成链式了。
✏️ 测验 1：以下哪个场景最适合用链式调用？

链式调用的工程实践

  1. 每步传递上下文 — 每一步的 output 作为下一步的 input，保持信息流不断
  2. 每步一个明确目标 — 不要在一个步骤里塞两个不相关的子任务
  3. 设计可检查的中间产出 — 每步的输出应该能被人工（或脚本）验证
  4. 考虑失败恢复 — 如果某一步失败了，是从头开始还是从这一步重试？
  5. 同一个 system prompt 贯穿 — 角色定义应该在所有步骤中保持一致
✍️ 练习：设计代码审查链

你需要审查一个 PR（包含 3 个文件的改动）。请设计一个 3 步链：代码审查 → 问题分类 → 汇总报告。写出每一步的 prompt 框架。

提示：Step 1 关注具体问题，Step 2 分类（bug/性能/风格），Step 3 输出汇总报告。
Step 1 — 审查（每个文件执行一次）：
<task>审查以下代码文件，列出所有问题。每条问题包含：文件路径、行号、问题描述。</task>
<file>（文件内容）</file>

Step 2 — 分类：
<task>将以下问题分为三类：BUG（可能影响运行）、PERF（性能问题）、STYLE（代码风格）。标注严重程度。</task>
<issues>（Step 1 的输出）</issues>

Step 3 — 汇总：
<task>写一份 PR 审查摘要。格式：修改概述 → 问题统计 → 必须修复项 → 建议项。</task>
<classified>（Step 2 的输出）</classified>

跨模型适用性

Prompt Chain 是完全通用的技术——它不依赖模型的任何特殊能力。所有模型都支持多轮对话，链式调用本质上就是精心设计的多轮对话。

总结

  1. 链式 = 复杂任务拆成多步，每步聚焦一个目标
  2. 自纠正链：生成 → 审查 → 优化，是最实用的链式模式
  3. 分解链：拆解 → 执行 → 组装，适用于大型分析任务
  4. 判断标准：需要中间检查、不同阶段需要不同指令 → 用链式
  5. 工程要点：传递上下文、单一目标、可检查产出

课后行动：从你日常工作中找一个多步骤任务（比如"翻译+审校+汇总"），拆成链式调用，对比单次 prompt 的效果差异。
