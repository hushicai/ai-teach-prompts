Few-Shot Prompting — 用示例驱动输出

Lesson 2 · 让模型通过例子学习 · 约 25 分钟

上一课我们学了 Anthropic 的 5 个核心原则。本课聚焦于其中最强力的技术之一：Few-Shot Prompting（示例学习）。 Anthropic 官方称其为 "one of the most reliable ways to steer Claude's output format, tone, and structure"。

► 官方参考：Anthropic — Use examples effectively

"Few-shot" 源自机器学习术语。Zero-shot = 不给例子直接做。Few-shot = 给少量例子让模型学。

为什么示例如此有效？

描述一个格式要求，不如直接给一个例子。原因是：

  • 例子消除歧义 — "用正式语气" 可以有一千种理解，但一个例子定死了
  • 例子展示边界 — 模型能从你给的例子中推断出 "这类输入对应这类输出" 的模式
  • 例子压缩知识 — 一个精心构造的例子比三段描述更节省 token
关键洞察：对于 LLM，展示 比 描述 更有效。Anthropic 将例子称为 "pictures worth a thousand words"。

好例子的三个标准

标准含义反例
相关 (Relevant) 贴近你的实际用例 做代码翻译却给文学翻译的例子
多样 (Diverse) 覆盖边缘情况和不同变体 3 个例子都是同一类型的输入
结构化 (Structured) 用 <example> 标签包裹，与指令区分 例子混在指令段落中

示例结构的最佳实践

使用 <examples> 包裹多个例子，每个例子清晰标注输入和输出：

<task>
将用户反馈分类为：投诉 / 建议 / 咨询 / 其他
</task>

<examples>
<example>
<input>你们的App太难用了，动不动就闪退</input>
<output>投诉</output>
</example>

<example>
<input>能不能加个夜间模式？晚上用太刺眼了</input>
<output>建议</output>
</example>

<example>
<input>请问你们的退款流程是怎样的？</input>
<output>咨询</output>
</example>

<example>
<input>天气不错</input>
<output>其他</output>
</example>
</examples>

请将以下用户反馈分类：
<input>登录页面加载太慢了，等了十几秒</input>
注意第四个例子是刻意加入的"边界情况"——一句无关的话应该被归为"其他"。边界例子能显著提升分类器的鲁棒性。
✏️ 测验 1：上面这个分类 prompt 的例子中，哪个例子最有价值（最体现"多样性"）？

示例的数量：多少最合适？

Anthropic 推荐 3–5 个例子。太少模型可能学不到模式，太多则浪费 token 且可能引入噪声。

例子的数量与任务复杂度之间的关系：

任务类型推荐示例数说明
简单分类（2–3 类）2–3 个每类一个即可
中等复杂度（多分类/格式转换）3–5 个覆盖主要变体和边界
复杂任务（代码生成/结构化输出）3–5 个优先质量而非数量

如果你不确定例子的质量和覆盖率，可以反过来让 AI 帮你评估：

请评估以下 3 个例子的相关性和多样性，识别缺失的边界情况，并补充 2 个新例子。
<examples>
...（你的例子）
</examples>

让示例与指令分离

一个常犯的错误是把示例混在指令里。模型需要能清晰区分"这是规则"和"这是例子"。

✗ 混在一起
请用 JSON 格式输出。比如 {"name": "张三", "age": 28} 这样。还有 {"product": "手机", "price": 3999}。请输出北京市的人口。
✓ 指令与示例分离
请以 JSON 格式回答，包含 name 和 value 字段。 <examples> <example>{"name": "张三", "value": 28}</example> <example>{"name": "iPhone 15", "value": 5999}</example> </examples> 北京市的人口是多少？
好处：分离后，你可以随时换例子而不改动指令部分。这是工程化的前提——prompt 的"代码"和"数据"解耦了。
✍️ 练习 1：为情感分类任务写示例

你已经写好了一条 prompt 的指令部分。请补充 4 个 <example>，覆盖"正面"、"负面"、"中性"和"边界情况"。

<task> 判断用户评论的情感倾向：正面 / 负面 / 中性 </task> <examples> <example> <input>（请填写）</input> <output>（请填写）</output> </example> ...（至少 3 个例子，覆盖不同倾向） </examples>
提示：边界情况可以是"这个产品还行"（中性），或者"如果服务能再好点就完美了"（负面）。
<examples>
<example>
<input>超级好用！一天不用就难受</input>
<output>正面</output>
</example>

<example>
<input>垃圾产品，再也不会买了</input>
<output>负面</output>
</example>

<example>
<input>今天下午三点送到</input>
<output>中性</output>
</example>

<example>
<input>还行吧，对得起这个价格</input>
<output>中性</output>
</example>
</examples>

关于「Few-Shot vs Multishot」的术语说明

Anthropic 官方文档现在统一使用 multishot prompting 来指代多个示例的技术（见官方文档"Use examples effectively"章节）。Few-shot 是学术界沿用下来的叫法，两者本质相同。本课程保留「Few-Shot」作为教学术语，但建议你在阅读官方文档时留意 multishot 这个关键词。

官方建议：将多个示例包裹在 <examples> 标签中，单个示例用 <example> 标签，这样 Claude 能更好地区分示例和指令。

跨模型适用性

Few-shot 是所有大模型通用的技术。区别在于：

模型对示例的敏感度说明
Claude高对 XML 标签中的示例响应最好
GPT-4o高同样受益于结构化示例，但可以用 markdown 代替 XML
Gemini中高示例有效，对格式一致性要求较高
DeepSeek / 国产模型中示例有效，但需要更简洁、更明确的格式

总结

  1. 展示胜过描述 — 一个精心构造的例子顶三段规则
  2. 3–5 个高质量示例 — 相关、多样、有边界情况
  3. 用 XML 标签包裹 — 让模型明确知道哪里是示例、哪里是指令
  4. 示例 = 数据，指令 = 代码 — 分离后易于维护和迭代
  5. 让 AI 帮你评审示例 — 用它来发现你遗漏的边界情况
课后行动：找一条你日常工作用的 prompt，看看能不能给它加上 3–5 个高质量示例。你会发现输出稳定性大幅提升。
