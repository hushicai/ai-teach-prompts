Context Engineering — 上下文工程

Lesson 10 · 跨会话状态管理 · 约 25 分钟

Anthropic 将Context Engineering定义为提示词工程的自然演进—— 从"写好单次 prompt"进化到"管理整个对话生命周期的信息流"。

当 Agent 需要长时间运行时，上下文会耗尽、信息会丢失、状态会混乱。 本课教你如何设计跨会话工作的 prompt 系统。

► 官方参考：Anthropic — Effective Context Engineering

Claude 4.6+ 具有 context awareness——能感知还剩多少 token 预算。你可以利用这一点让它在上下文耗尽前保存状态。

问题：上下文窗口不是无限的

即使模型支持 200K token，长期运行的任务会面临：

  • 上下文耗尽 — 对话太长，早期信息丢失
  • 注意力稀释 — 关键指令淹没在历史对话中
  • 状态混乱 — Agent 忘记做了什么、做到哪了

解决方案不是"换更大的上下文窗口"，而是主动管理上下文。

策略一：结构化状态文件

把关键状态写入外部文件，新会话开始时读取：

// 结构化状态文件 — state.json
{
  "session": 3,
  "completed_tasks": ["authentication_flow", "user_management"],
  "current_task": "api_endpoints",
  "remaining": ["notification_service", "reporting"],
  "known_issues": [
    { "task": "user_management", "issue": "test #42 偶发失败", "status": "investigating" }
  ]
}

// 进度笔记 — progress.txt
Session 3 progress:
- Fixed auth token validation
- Updated user model
- Next: investigate test #42 failure
- Note: do not remove tests

在新会话开始时：

请从这些文件中恢复工作状态：
1. 读取 progress.txt 了解进度
2. 读取 state.json 了解任务列表
3. 读取最近的 git log 了解已完成的变更
然后继续当前任务。

策略二：上下文感知

Claude 4.6+ 能感知剩余 token。你可以利用这个能力：

<context_management>
你的上下文窗口将在接近上限时自动压缩，你可以在压缩前保存进度。

因此：
- 不要因为 token 预算问题提前中止任务
- 当你接近 token 上限时，将当前进度和状态保存到 progress.txt 和 state.json
- 在上下文刷新后，从这些文件中恢复
- 不要人为提前结束任务
</context_management>

策略三：首次上下文设计

跨会话工作的第一个上下文窗口最重要——它建立框架：

// 第一个会话的 prompt
这是这个项目的第一个工作会话。请：
1. 分析项目结构，创建一份状态文件（state.json）跟踪进度
2. 为每个模块创建测试框架
3. 创建 setup.sh 脚本，简化后续的环境配置

在后续会话中，你需要：
1. 读取 state.json 恢复进度
2. 运行 setup.sh 恢复环境
3. 继续按 state.json 中的列表工作
✏️ 测验 1：跨会话工作最关键的设计是什么？

策略四：Git 作为状态层

Anthropic 推荐将 git 作为跨会话的状态管理工具：

  • 提交记录 = 进度日志 — 每个提交记录了完成的工作
  • diff = 当前变更 — 新会话可以从 git diff 了解未完成的工作
  • 分支 = 探索路径 — 不同方案可以并行尝试
  • 可回滚 — 出错了可以安全恢复

在新会话中：

请先运行 git log --oneline -10 了解最近完成的工作，
然后运行 git diff --stat 了解当前的未完成变更，
再继续工作。

策略五：启动脚本

在第一个会话中创建恢复脚本，后续会话直接使用：

// init.sh
#!/bin/bash
echo "=== 状态恢复 ==="
cat state.json
echo "=== 进度 ==="
cat progress.txt
echo "=== 最近提交 ==="
git log --oneline -5
echo "=== 测试状态 ==="
python -m pytest --summary 2>/dev/null || echo "测试框架尚未配置"

在新会话中：

运行 init.sh 恢复工作状态，然后继续 progress.txt 中记录的任务。
✍️ 练习：设计跨会话状态恢复流程

你正在开发一个为期 5 天的数据分析项目。请设计一个 prompt，告诉 AI 如何在新会话中恢复工作状态。

提示：告诉 AI 读取哪些文件、运行什么命令、如何检查进度。
欢迎回来。请按以下步骤恢复工作状态：

1. 运行 bash init.sh 查看当前进度
2. 读取 state.json 了解任务清单和完成情况
3. 读取 progress.txt 了解详细进度和已知问题
4. 运行 git log --oneline -10 检查最近的提交
5. 找出 state.json 中第一个未完成的任务并开始工作

重要规则：
- 不要重复已完成的工作
- 如果发现之前的任务标注为 completed 但实际未完成，更新 state.json
- 每个任务完成后更新 state.json 和 progress.txt

跨模型适用性

上下文工程是跨模型通用的方法论。所有主流模型（Claude、GPT-4o、Gemini、DeepSeek）都支持文件读取、状态管理和多会话工作流。唯一差异在于国产模型的上下文窗口较小，建议精简状态文件。

总结

  1. Context Engineering 是提示词工程的进化——管理整个生命周期的信息流
  2. 结构化状态文件 — JSON 跟踪任务、纯文本跟踪进度
  3. 上下文感知 — 利用模型的 token 感知能力自动保存
  4. 首次上下文 — 第一个会话建立框架，后续会话恢复
  5. Git 即是状态 — 提交记录是最可靠的状态层
  6. 启动脚本 — init.sh 一键恢复
