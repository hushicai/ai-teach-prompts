Anthropic 的提示词工程方法论建立在几个简单但强大的原则上。 无论你用哪个模型——Claude、GPT、Gemini 或是国产模型——这些基本原则都适用。 一个结构清晰、意图明确的 prompt，远比一段冗长无序的描述有效。

► 完整官方指南：Anthropic Prompting Best Practices

1. 黄金法则

Anthropic 官方原文："Think of Claude as a brilliant but very new employee (with amnesia) who needs explicit instructions."

Anthropic 提示词工程的核心理念只有一句话：

把 AI 当作一个聪明但第一天上班的实习生——没有上下文、没有常识，但能完美执行清晰的指令。

这意味着：

  • 不要假设 AI "知道"你的业务背景 — 它不知道你的团队、你的代码库、你的用户
  • 不要假设 AI "能猜出"你想要什么 — 如果你想要"超出预期的效果"，明确说出来
  • 不要用模糊的请求 — "分析这份报告" 不如 "分析这份报告，列出三个关键风险，每个风险给出缓解建议"
💡 自检技巧：写完 prompt 后，把它给一个不了解这个任务的同事看。如果他会困惑，那 AI 也会。

2. 清晰直接

最容易被低估的技巧：直接说你要什么。AI 不需要"礼貌"或"试探性"的语言——它需要精确的指令。

✗ 模糊版
帮我看看这段代码，有什么问题吗？
✓ 清晰版
审查以下 Python 函数，找出： 1. 潜在的 bug 2. 性能瓶颈 3. 类型安全问题 对每个问题标注严重程度（高/中/低）。
关键区别：模糊版让 AI 自己决定"什么问题"——它可能给出泛泛的回答。清晰版限定了范围（bug/性能/类型安全）和格式（标注严重程度），结果更有用。

3. XML 标签：结构化输入

这是 Anthropic 的标志性技巧。Claude 在训练过程中大量使用 XML 数据，因此天然擅长解析 <tags>。

当 prompt 包含多种类型的内容时（指令、数据、例子、输出格式），用 XML 标签将它们分隔开。这不仅让 prompt 更可读，也让 AI 更精确地理解每部分的作用。

<task>
将以下英文邮件翻译成中文，保持正式商务语气。
</task>

<email>
Dear Team, I would like to inform you that the Q3 budget has been approved.
</email>

<requirements>
- 不使用"亲爱的"作为开头
- 署名统一用"此致"
</requirements>

标签命名规范：

  • 用有意义的英文名称：<instructions>、<context>、<examples>、<output>
  • 保持一致性：同一个概念始终用同一个标签名
  • 支持嵌套：文档集可以放在 <documents> 中，每篇文档用 <document>

下面是一个"无结构"的 prompt。请用 XML 标签重新组织它：

请帮我写一个 Python 函数，它接受一个 URL 列表，下载每个 URL 的内容，然后返回一个字典，key 是 URL，value 是内容。要用 requests 库，处理超时和异常。每个 URL 超时设为 10 秒。
提示：考虑用 <task>、<requirements>、<libraries>、<output_format> 等标签
<task>
写一个 Python 函数，接受 URL 列表，下载每个 URL 的内容，返回 URL → 内容的字典。
</task>

<libraries>
使用 requests 库
</libraries>

<requirements>
- 处理连接超时和请求异常
- 每个 URL 超时设置为 10 秒
</requirements>

<output_format>
dict[str, str] — key 是 URL，value 是下载到的内容文本
</output_format>

4. 角色设定

Anthropic 官方用一个简单示例展示了角色设定的威力：一行 "You are a helpful coding assistant" 就足以改变输出风格。

给 AI 一个明确的角色，会在系统层面调整它的输出风格、关注点和专业程度。

系统提示词（System Prompt）：
你是一名资深 Python 后端工程师，精通性能优化和代码审查。
在给出建议时，始终附带时间复杂度分析和内存占用估算。

好的角色设定应该包含：

  • 专业领域（资深 Python 后端工程师）
  • 核心职责（代码审查、性能优化）
  • 行为约束（始终附时间复杂度分析）

5. 上下文：解释"为什么"

Anthropic 发现，告诉 AI 为什么 某些约束存在，比简单地说"不要做 X"效果好得多。AI 能从这个原因泛化到你没预料到的边界情况。

大多数人在写 prompt 时只告诉 AI "要做什么"——但告诉它"为什么"同样重要。当 AI 理解了背后的原因，它就能在遇到未明确说明的情况时做出正确判断。

✗ 只有规则
不要用省略号。 不要用表情符号。 不要用被动语态。
✓ 规则 + 原因
你的输出将通过 TTS 引擎朗读。 因此： - 不要用省略号（TTS 无法正确朗读） - 不要用表情符号（TTS 会忽略它们） - 优先用主动语态（主动语态更自然）

给上下文的方式：

  • 这个输出的用途是什么？（面向客户？内部文档？）
  • 目标受众是谁？（CEO？工程师？小白用户？）
  • 这个任务在整个工作流中的位置？（前置调研？最终输出？）
  • 成功的标准是什么？（什么算"好"？）
✍️ 练习 2：添加上下文

下面这条 prompt 没有上下文。请添加一段上下文说明，让 AI 知道：输出将被展示给非技术背景的客户，目的是说服他们采纳你的技术方案。

总结与下一步

这 5 个原则构成了 Anthropic 提示词工程的基础。你可以把它们当作写任何 prompt 前的检查清单：

  1. 黄金法则 — 当作第一天上班的实习生
  2. 清晰直接 — 明确说你想要什么，不想要什么
  3. XML 标签 — 用结构化分隔指令、数据、输出格式
  4. 角色设定 — 给 AI 一个专业身份和行为约束
  5. 解释原因 — 告诉 AI 为什么，让它帮你应对边界情况
🔗 跨模型适用性：这 5 个原则是所有大模型通用的。Claude 对 XML 标签特别敏感，但 GPT、Gemini、DeepSeek 等模型同样能从结构化 prompt 中受益。角色设定和上下文解释更是通用技巧。
